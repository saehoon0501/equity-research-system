## Expected Behavior per Architectural Lock

Per the v3 spec architectural locks, the pipeline must:

### Activation gate (Section 4.5 Q6)

`abs(drawdown_vs_benchmark_pp) >= MODE_2X_THRESHOLDS[mode]` — for Mode C this
is 30pp. With drawdown -65pp the gate is satisfied and the pipeline runs.
Below 30pp the pipeline must short-circuit with
`cut_status='not_activated_below_2x_threshold'` and emit no DB writes.

### Layer 1 — cooling-off floor (Section 4.5 Q6 Layer 1)

24h has elapsed since trigger; `evaluate_cooling_off` returns `blocking=False`
and the pipeline proceeds to Layer 2.

### Layer 2 — multi-source confirmation (Section 4.5 Q6 Layer 2)

Two independent kill fires (no shared BOCPD correlation group), one carries
a verbatim primary quote sourced from a 10-K, and the premortem lookup
returns True (a premortem within 30 days exists). All three sub-checks pass;
`independent_kill_count = 2`, `all_satisfied = True`.

### Layer 3 — counterfactual VETO authority (Section 4.5 Q6 Layer 3)

Top-3 retrieved by mechanical scoring against the 14-case catalog (active
pool only). With PLTR's SURVIVOR-shaped universal core, the top-3 must
include ≥2 SURVIVOR-leaning cases (SURVIVOR + DILUTED-SURVIVOR), forcing
`status='operator_override_required'` and blocking the auto-cut.

### Persistence + alert (Sections 5.3 + 6 Q6 PB#5)

* `counterfactual_retrievals` row written (always — every pipeline run logged).
* `veto_lifecycle` row written when veto invoked.
* `unread_alerts` row inserted when SURVIVOR-dominant
  (`alert_type='counterfactual_veto'`, `severity=3`).

### HMAC tamper-evidence (Section 7.2 invariant)

The catalog rows feeding Layer 3 must be HMAC-verified at load time. Any
row whose stored `hmac_signature` does not match the canonical payload is
dropped from the active pool, with a `system_errors` row emitted
(`source='counterfactual_veto.retrieval'`, `error_type='peak_pain_hmac_invalid'`).
Without this gate, a tampered SURVIVOR row could be flipped to NON-SURVIVOR
features and silently turn the blocking veto into a non-blocking decision.

The discrimination is real: AAPL-2003 (same sector) gets the +0.3 sector-

entirely. Bayesian shrinkage caps universal-core similarity at 0.75 even
on a 6/6 match — no similarity is exactly 1.0. This contrasts with the
prior smoke-test fixture, where 3 hand-picked cases were identical to the
candidate; here, the SURVIVOR-dominant top-3 is earned by the scorer
recognizing PLTR's structural similarity to historical SURVIVOR cases
across sector boundaries.

## Verdict

**OPERATOR-OVERRIDE-REQUIRED.** The candidate features at the 2022 trough
match historical SURVIVOR cases (NVDA-2008, NFLX-2011, MELI-2022, CVNA-2022,
AAPL-2003) on universal-core 6/6. Mechanical retrieval surfaces SURVIVOR-
dominant top-3 across the realistic catalog. Per Section 4.5 Q6 Layer 3,
the auto-cut is blocked; the operator must explicitly override (capturing
rationale via `lifecycle.operator_override`) before any cut executes.

This is the PLTR-2022 problem: an algorithm that auto-cuts on Mode-C
2×-threshold drawdowns alone would have liquidated PLTR at the 2022 trough,
realizing the ~65pp drawdown and forfeiting the subsequent recovery. The
counterfactual VETO turns that liquidation into a deliberate operator
decision.

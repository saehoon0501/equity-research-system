**Date:** 2026-05-21

## Decision

**Split BUY → BUY-HIGH / BUY-MED in the tactical_disposition mapping.**

Updated mapping table (replaces Section 2 v3-final's Plan C v5 mapping):

| Conviction \ Tactical | Negative | Neutral | Positive | Unavailable |
|---|---|---|---|---|
| HIGH | HOLD | HOLD | **BUY-HIGH** | HOLD |
| MEDIUM | HOLD | HOLD | **BUY-MED** | HOLD |
| LOW | AVOID | AVOID | AVOID | HOLD |

## Why split (load-bearing argument)

**Renderer-compact-surface preservation.** Section 2 v3 specifies pm-supervisor's final report (full envelope view) but the renderer contract for downstream surfaces (daily digests, alert payloads, audit-trail one-liners, counterfactual_ledger entries) is Section 3 design — NOT enforceable from Section 2.

Single-BUY labels would require those compact surfaces to ALWAYS co-surface conviction + cell_size_pct alongside disposition to preserve information. BUY-HIGH / BUY-MED labels encode conviction IN THE LABEL, robust to whatever compact-surface decisions Section 3 makes.

The argument holds IFF Section 3 renderer contract permits conviction-dropped compact surfaces. If Section 3 mandates ALWAYS co-surfacing conviction, the split's load-bearing rationale weakens and Phase 2 re-merge applies (see below).

## Why BUY-HIGH / BUY-MED (label choice)

Reviewer iteration 2 caught "BUY-strong / BUY-light" as carrying pejorative connotation ("light = less serious"). v3 adopted BUY-HIGH / BUY-MED instead:
- Literally encodes the conviction tier (HIGH or MEDIUM)
- No magnitude language; no connotation asymmetry
- Mirrors the conviction column literally; zero semantic guessing
- Symmetric with existing HOLD / AVOID labels (no qualifiers)

## INV-2.1-A (disjointness invariant)

`summary_code` enum (BUY/HOLD/TRIM/SELL — canonical, per `tests/test_envelope_shape.py` Consensus Item #1) and `tactical_disposition` enum (BUY-HIGH/BUY-MED/HOLD/AVOID) are DISJOINT. No derivation rule between them.

**Section 3 implementation MUST enforce via:**
- Separate HG validator regex for tactical_disposition (NOT reuse of `^(BUY|HOLD|TRIM|SELL)$` from summary_code validation)
- Separate database columns / JSONB keys for the two fields
- Parallel test assertion in `tests/test_envelope_shape.py`: `for code in ("BUY-HIGH", "BUY-MED", "HOLD", "AVOID"):` distinct from canonical summary_code assertion

If Section 3 violates INV-2.1-A, the Phase 1 step 8 enum-validity correctness check catches it.

## Phase 1 acceptance (Section 2 v3 step 8 — tightened by Section 2.1)

Per-label fire-rate audit tracks 5 labels at Section 2.1 (was 4 at Section 2 v3):
1. downward OVERRIDE
2. upward DIVERGENCE
3. comparator HOLD-active (BUY-HIGH subcase)
4. comparator HOLD-active (BUY-MED subcase)
5. LOW-CONVICTION VETO

**Hard gate:** all 5 enum values appear as VALID outputs of the renderer logic across the cohort. n=12 may produce 0 of any individual label — that's OK; the check is that WHEN the label fires, it conforms to the disjoint enum per INV-2.1-A. Catches Section 3 schema bugs (e.g., HG validator rejecting BUY-HIGH because it was reused from summary_code regex).

**Fire-rate logging:** per-label counts logged for Phase 2 baseline; NO threshold-setting at Phase 1 (n=12 too small for statistical inference; BUY-HIGH expected <1 obs given 17% HIGH conviction rate × P(positive bin)).

## Phase 2 decision matrix (4 quadrants)

Phase 2 trigger fires at envelope_count ≥ 50 AND ticker_count ≥ 5 per Section 2 v3 auto-review trigger.

| Returns spread (BUY-HIGH vs BUY-MED 12mo realized) | Operator-ack rate difference | Phase 2 action |
|---|---|---|
| ≤ ±1σ | ≤ ±10% | **Auto-re-merge** to single BUY (2-row UPDATE; symmetric cost to split) |
| ≤ ±1σ | > ±10% | **Phase 3 deferral** (log; do NOT auto-decide) |
| > ±1σ | ≤ ±10% | Keep split |
| > ±1σ | > ±10% | Keep split (strong evidence) |

The mergeable-returns / divergent-ack quadrant cannot be auto-decided at Phase 2 — operator ack divergence might reflect label CLARITY (operators correctly route attention) or label CONFUSION (operators mechanically over-trust suffix). Phase 3 may discriminate.

## Phase 3 deferral with 18-month deadline

For (returns ≤1σ AND ack >10%) Phase 3 deferral:
- **Evaluation window:** counterfactual_ledger 12mo realized accumulation
- **Default deadline:** 18 months from Phase 2 trigger date
- **Default action at deadline:** if Phase 3 cannot distinguish label-clarity vs label-confusion, **default to re-merge to single BUY**
- **Reasoning for default:** keeping the split has ongoing MAINTENANCE-SURFACE cost (label surface area, operator cognitive load, renderer branching — maintenance/cognitive, NOT runtime/memory); re-merging is symmetric-cost reversible if later evidence emerges; defaulting to merge avoids indefinite-purgatory failure mode.
- **Early decision allowed:** Phase 3 can decide before deadline if counterfactual evidence emerges (e.g., operators trade HIGH-suffix BUYs at materially different sizes with proportionate alpha).

Re-merge UPDATE: `UPDATE parameters SET value='BUY' WHERE parameter_key IN ('tactical_disposition.mapping.HIGH_positive', 'tactical_disposition.mapping.MEDIUM_positive')` + revert enum extension.

Worst-case timeline: Phase 2 trigger may take 12+ months to accumulate cohort; Phase 3 18-month deadline runs FROM Phase 2 trigger, so total persistence can reach 30+ months. v5 accepts this — clock-anchoring to Phase 2 (not Section 2.1 lock) preserves Phase 3's evaluation validity (Phase 3 requires populated cohort).

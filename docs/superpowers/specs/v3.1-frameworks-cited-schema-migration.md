# frameworks_cited Schema Migration — Dual-Read Shim

**Status:** Implementation IMPLEMENTED + tested. Awaits /review-me before merge.
**Parent:** /grill-me Section 1 consensus CAF-2 (`8441a32`) + Phase 0 enum v3.1 (`3f38bab`).

---

## Purpose

Migrate the `frameworks_cited` field schema from **list-of-entries** to **keyed-object** form so that HG-29 Check 4d's restricted path grammar can resolve via named-key syntax (e.g. `frameworks_cited.mauboussin_reverse_dcf.output.implied_growth_pct`) rather than brittle integer indices (`frameworks_cited.0.output.x`).

**The bug this surface closes:** Phase 0 enum's HG-29 Check 4d locked path grammar requires deterministic resolution. Integer indices are determinism-fragile — index 0 might be damodaran_narrative_dcf in one envelope and buffett_2007_inevitables in another, depending on emit order. Named-keys are stable.

---

## Strategy: dual-read shim, soft migration

Per consensus CAF-2 (which warned of 7+ downstream consumer blast radius), implementation uses a dual-read shim accepting BOTH schemas indefinitely:

- **Legacy list form (pre-v3.1):**
  ```yaml
  frameworks_cited:
    - framework_key: damodaran_narrative_dcf
      output: {...}
    - framework_key: buffett_2007_inevitables
      output: {...}
  ```

- **New keyed-object form (v3.1+):**
  ```yaml
  frameworks_cited:
    damodaran_narrative_dcf:
      framework_key: damodaran_narrative_dcf
      output: {...}
    buffett_2007_inevitables:
      framework_key: buffett_2007_inevitables
      output: {...}
  ```

The shim's API surface (`src/evaluator_gates/_frameworks_cited_shim.py`):

| Function | Purpose | Dual-read? |
|---|---|---|
| `find_framework(memo, key)` | Resolve entry by framework_key | YES |
| `iter_frameworks(memo)` | Iterate all entries in insertion order | YES |
| `get_framework_keys(memo)` | Return set of framework_keys present | YES |
| `is_keyed_object_form(memo)` | True iff v3.1+ form (migration telemetry) | n/a (the discriminator) |

## Path-grammar resolution for HG-29 Check 4d

HG-29 Check 4d locks the path grammar: "array access via explicit integer index OR canonical framework_id (named-key on `frameworks_cited`). No wildcards, no filters."

Both forms remain resolvable:
- `frameworks_cited.0.output.implied_growth_pct` resolves via legacy list-form (index 0)
- `frameworks_cited.mauboussin_reverse_dcf.output.implied_growth_pct` resolves via keyed-object form

The path resolver MUST be implemented in evaluator's HG-29 Python harness when that lands (currently HG-29 Check 4 is described in evaluator.md prose; Python implementation deferred). When Python harness lands, it uses the shim's `find_framework(memo, framework_key)` for named-key paths and direct indexing for numeric paths.

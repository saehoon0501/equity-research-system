## Section 4 — Architectural Decision Locks Captured

Each lock below is enforced by at least one assertion in the test suite. Spec citations are given in the v3 spec (`docs/superpowers/specs/2026-04-29-empirical-foundation-design-v3.md`).

1. **BOCPD dual-signal architecture** — canonical marginal pinned near hazard rate in steady state; `short_run_mass` is the firing signal. v3 §2.1 + migration 020. Tests: `test_regime_sidecar.py::test_dual_signal_*`.

2. **Conviction rollup precedence: LOW > HIGH > MEDIUM** — LOW dominates if any pillar is LOW; otherwise HIGH wins over MEDIUM. v3 §4.7. Tests: `test_p7_recommendation_emitter.py::test_*conviction_rollup*`.

3. **HIGH gate monotonic interpretation (≥2 SURVIVOR; mixed acceptable)** — counterfactual-veto requires ≥2 SURVIVOR matches but does NOT require all 3 to be survivors. v3 §4.4 Q3. Tests: `test_counterfactual_veto.py::test_2_survivor_*` + `test_p7_recommendation_emitter.py::test_2_survivor_1_non_survivor_with_high_signals_resolves_to_HIGH`.

4. **Hysteresis 2-cycle persistence with pending_target reset semantics** — bucket transitions require 2 consecutive cycles agreeing on the new target; intervening disagreement resets the pending_target counter. v3 §4.7 + §4.6 Q4. Tests: `test_p7_recommendation_emitter.py::test_hysteresis_pending_target_change_starts_fresh_cycle_count` + `test_conviction_flip_flop_walkthrough.py::TestFlipFlopSuppression::*`.

5. **MAX_EMAIL_ATTEMPTS = 4** — full 1m / 5m / 15m / final retry schedule per Phase 4 Q9. v3 §5.3. Tests: `test_alert_channels.py::test_max_attempts_*`.

6. **Canonical-payload HMAC contract (4 distinct env-var scopes)** — `AUDIT_HMAC_KEY`, `PEAK_PAIN_HMAC_KEY`, `PREMORTEM_HMAC_SECRET`, `WATCHLIST_HMAC_KEY` — never share keys across scopes; each independently revocable. Canonical payload is JSON with `sort_keys=True, separators=(',',':'), ensure_ascii=False`. v3 §5.2 + §7 Q4 + migration 016. Tests: `test_audit_trail.py` + `test_hmac_integration.py::test_canonical_payload_*`.

7. **Anchor-drift forced_review pending state via sidecar table** — pending state lives in the migration-018 sidecar, NOT in the watchlist row itself, to preserve immutable-pillar HMAC. v3 §4.6 Q3. Tests: `test_anchor_drift.py::test_*pending*`.

8. **mode_certainty as separate annotation, not conviction-bucket determinant** — mode classification feeds disposition view + horizon weighting but does NOT collapse into the conviction bucket. v3 §2.2 Q-lock. Tests: `test_mode_classifier.py::test_mode_certainty_does_not_override_conviction*`.

9. **Sonnet/Opus only — NO Haiku in spec-locked decision paths** — Section 4.5 Q1 model constraint applies to debate, daily-monitor classification, premortem devil's-advocate, and peak-pain catalog. Verified by code-grep + test fixtures (no `claude-haiku-*` model strings present in `src/p3_mechanical_scorer/`, `src/p4_debate/`, `src/l4_daily_monitor/`, `src/peak_pain_catalog/`, `src/premortem_scheduler/`).

## Section 8 — How Operator Launches v0.1

Step-by-step launch checklist. Read `docs/superpowers/operator-reference.md` first for command-by-command detail.

### Step 1 — Generate HMAC keys (4 scopes)

```
python -c "import secrets; print(secrets.token_urlsafe(32))"   # × 4, one per scope
```

Set in `.env` (see `.env.example`):
- `AUDIT_HMAC_KEY` — execution_recommendations + audit_provenance row signing.
- `PEAK_PAIN_HMAC_KEY` — peak_pain_archetypes row signing.
- `PREMORTEM_HMAC_SECRET` — premortem row signing.
- `WATCHLIST_HMAC_KEY` — watchlist pillar signing.

Never reuse the same key across scopes — each is independently revocable per v3 §5.2.

### Step 2 — Set SMTP credentials

In `.env`, populate `ALERT_SMTP_HOST`, `ALERT_SMTP_PORT`, `ALERT_SMTP_USER`, `ALERT_SMTP_PASSWORD`, `ALERT_SMTP_FROM`. The 4-attempt M-2/M-3 retry schedule (1m / 5m / 15m / final) requires working SMTP for non-degraded behaviour. Per v3 §5.3 + Phase 4 Q9.

### Step 3 — Broker MCP authorize

```
python -m src.mcp.broker_mcp.oauth authorize
```

Browser-interactive — opens the Schwab consent flow, captures the redirect, populates token fields in `.env`. Schwab refresh tokens last 7 days; if idle >7d, re-auth.

### Step 4 — Peak-pain catalog priority subset

```
python -m src.peak_pain_catalog.cli priority-run
```

Runs the 3-LLM consensus across the priority subset. Operator reviews HIGH consensus ≥95% rows; signs them with `PEAK_PAIN_HMAC_KEY`. Disputed rows roll to v0.5+ tail validation.

### Step 5 — Materiality classifier kappa calibration

Operator rates 30 historical events (material/immaterial), system computes Cohen's kappa vs rule output, and records the result. Threshold: kappa ≥0.61. The harness lives in `tests/test_l4_daily_monitor.py::test_cohens_kappa_*`.

### Step 6 — Per-gate sign-off (33 gates)

For each launch gate listed in v3 §7.3 + §7.3a:

```
/launch-confirm <gate_name> [--note "<short context>"]
```

Each invocation appends an HMAC-signed row to `docs/superpowers/launch-readiness-log.md` (append-only).

### Step 7 — Verify and flip

```
/run launch-gates
```

When the report shows 33/33 → v0.1 is ready for active emission mode. Per BUILD_LOG decision 5, `/run` auto-detects phase based on gate state.

## Appendix K — Post-launch sanity-check protocol

Within 72 hours of flipping to active emission mode, the operator should run the following sanity checks. Each is a soft check (not a launch gate); the goal is to catch silent failures early.

1. **Emission cadence sanity.** After 2 full daily cycles, run `/disposition` and confirm at least one row has updated since launch. If not, the daily cadence may be silently no-oping; check `system_errors` last 7 days.

2. **HMAC chain end-to-end check.** Pick one recent `execution_recommendations` row; run `/audit-trail <rec_id> --verify --strict`. Should return VERIFIED. If TAMPER-EVIDENT, escalate to M-2 incident.

3. **Counterfactual retrieval sanity.** For a recently-emitted name with HIGH conviction, run `/audit-trail <rec_id> --stage stage_3_counterfactual`. Confirm exactly 3 retrieval results, ≥2 marked SURVIVOR (per HIGH gate monotonic interpretation).

4. **Anchor-drift pending state surface.** Run `/disposition --mode B`; mode-fit dashboard at the bottom should render without error. If `pending_reclassification` flag is present unexpectedly, investigate via `/audit-trail`.

5. **Alert delivery loop.** Trigger a test M-2 alert (operator-only path; documented in `docs/superpowers/operator-reference.md` Section 6). Confirm email arrives within 1 minute (first attempt). If retry schedule trips, MAX_EMAIL_ATTEMPTS=4 path activates.

6. **Premortem cadence.** Confirm at least one premortem-due row appears in `premortem_due` view within the first 30 days for any B/B'/C-mode name. Cadence is mode-tuned: B=180d, B'=120d, C=60d.

7. **System health.** Run `/system-health` daily for the first week. Watch for degraded MCPs, queue depth growth, push-alert backlog.

If any of the 7 checks fails repeatably, do NOT continue active emission — revert to passive build mode by toggling the orchestrator phase flag and investigate. The `parameters_runtime` table holds the runtime configuration and can be edited operator-side without redeployment.

# Requirements Document

## Project Description (Input)

**Feature:** `in-session-monitor` — a scheduled in-session LLM supervisory loop for the reactive CFD execution layer. (Full discovery record: `.kiro/specs/in-session-monitor/brief.md`; source of record: `docs/exploration-systematic-flow-architecture-2026-05-28.md` §15, with §14.7/§14.8/§14.10/§14.11.)

**Who has the problem.** The operator running the paper (→ pre-live) reactive CFD book. The committed build pushes **autonomous LLM authority into the live levered session** (§15), stacked on **autonomous between-session promotion** (§14.11 #2) — the §11.5 kill switch + deterministic survival gate are the *entire* backstop. That deterministic reflex only catches the **hard-survival** band (margin-distance breach); the after-market `walkforward-tuning-loop` catches behavioral drift but only at **weeks-apart** walk-forward boundaries (§14.6, PSR/MinTRL-sized).

**Current situation.** `decision-trace-telemetry` has landed the model-process trace + the **4 correlation keys** (`run_id, code_version, param_version, walk_forward_window`) and the append-only reader surface (mig 048); calibration (Brier/reliability) is the named behavioral diagnostic (§14.7). `execution-daemon` (requirements + design **approved 2026-05-30**) is the monitored process and exposes the **command surface** this loop commands — three gated seams (`engage-kill-switch` / `set-safe-mode-grade` / `select-validated-config`), apply-only-through-gated-paths, reject-direct-mutation (Req 9.3), reflect-tightened-but-never-self-loosen (Req 7.4), reflex-first (Req 7.3), adopt-via-atomic-hot-swap (Req 9.4); the daemon — not this loop — performs the hot-swap, so the monitor *selects* and the daemon *deploys*. `walkforward-tuning-loop` (requirements generated) **populates + OOS-validates** the config menu and writes it to the **P2 param-version registry** the monitor selects from. The daemon's `execution_daemon_event_queue` (mig 051) is drained **after-market by the tuner** (single `drained_at` setter), not by this monitor. §14.3 originally said "no LLM in-session" — **overturned by §15** for *monitoring + intervention* (fitting stays after-market). Nothing yet reads telemetry intra-session to catch **sub-survival drift** (e.g. softmax calibration breaking down while the model is still inside survival limits) and intervene; a drifted model keeps firing at retail latency for weeks.

**What should change.** Stand up a **distinct scheduled Claude Code orchestration spec** (markdown per P1; sibling to `walkforward-tuning-loop`; never daemon-dispatched per §14.10) that, on a regular in-session cadence: (1) **reads** recent decision-trace telemetry + the calibration diagnostic; (2) **judges** whether the model is behaving inside its calibrated envelope — flags **derived** from Brier/reliability, never asserted (P15); (3) **intervenes when it isn't** by commanding **only existing deterministic mechanisms** — kill switch (§11.5) / reproducible safe-mode (§14.3) / versioned-config-select (§14.5/P2) — **never a new direct-mutation path** (§15 central invariant); and (4) **audits** every intervention with a falsifiable rationale (its own envelope + HG validator, P11), correlated to the model trace via the 4 shared keys.

**Authority is operator-locked to §15 reading #2:** (a) *operational* — halt / flatten / restart a wedged component, clear bad state; **and** (b) *apply a pre-validated config* — select among already-validated param sets / tighten to a safer known version, then resume. It **never fits new values live** (reading #3 rejected — no out-of-sample exists intra-session, §14.4). It **selects from** the validated menu the tuning loop **populates**; it never fits or validates (the two-loop separation, §15).

**Invariants the requirements must enforce.**
- **Reflex-first ordering** — the deterministic Survive reflex (survival-gate) fires first; this LLM is the slower deliberative **second line** and never blocks survival on a round-trip (§13 / §15).
- **P7** — interventions are only ever *more* conservative (halt / tighten / select-safer); never loosen.
- **Commands-into-existing-mechanisms only** — no new mutation path (§15).
- **P11** — owns its own intervention-audit envelope + HG validator (the *why* of intervention lives here, not in `decision-trace-telemetry`, per §14.8).
- **P15** — drift/anomaly judgments derived from calibration; no performative `P(X)≈N%`.
- **T4 / §14.11 #4** — **no aggregate cost cap**; the monitor cadence has no aggregate halt → cadence must be set conservatively (eyes-open risk-acceptance).
- **v0.1 paper-only** (§11.5 / §16.1) — full in-session autonomy acceptable only in paper/challenger; before any live real-money cutover the gates + kill switch must be proven green (P14 + §11.5).

**Explicitly out of scope.** Fitting/validating any new param/code version (`walkforward-tuning-loop`); the hard-survival deterministic reflex itself (`survival-gate` + daemon); owning the kill-switch / safe-mode / config-select *mechanisms* (it commands them, owns none); the model-trace schema + correlation keys (`decision-trace-telemetry`); order-trigger / fill mechanics (`execution-daemon` / `broker-cfd-adapter`); gap-risk / real-time per-instrument halt detection (`survival-gate` domain, out of boundary per R7).

**Open items to resolve during requirements/design (not blockers).**
- **⚠️ Command transport (biggest unresolved seam).** The daemon exposes its command surface as **in-process `commands.py` seams** (daemon Req 9.2/9.3), but the monitor is an **out-of-process markdown orchestration** and the daemon is a **persistent non-MCP process** (§14.10) — so *how a command reaches the daemon* is undefined. kill-switch/safe-mode plausibly ride **shared state read fresh each loop** (the op-state freshness channel the Operator's halt uses); `select-validated-config` (force a specific safer, non-latest version) has **no defined inbound channel** (the daemon adopts *latest-published* at hot-swap) and likely needs a daemon-owned command-intake table. Resolve at design, coordinating an `execution-daemon` follow-up.
- **Trigger model** — fixed-interval scheduled cadence (inferable default) vs. a hybrid that also consumes daemon-emitted anomaly events. ⚠️ Now concrete: the daemon's `execution_daemon_event_queue` (mig 051) is the **tuner's** after-market drain target (single `drained_at` setter) — the monitor must **not** become a second drainer; a hybrid needs a read-only non-draining view or a separate signal.
- **Audit store + migration number** — if the P11 intervention-audit is a DB table, claim **≥ 054** (048 / 049–050 / 051–052 / 053+ taken; coordinate with `walkforward-tuning-loop`). The audit owns the **why** only; the **what** of each command is already emitted by the daemon as a `command`/`safe_mode`/`kill_switch` event (join via the 4 keys).
- **`select-validated-config` direction-of-change** — does the daemon enforce *toward-safer-only* on config selection, or does the monitor carry that P7 obligation alone? Confirm against the daemon `commands` seam at design.
- **P2 param-version registry shape** — the "menu" the monitor selects from is not yet a pinned schema (deferred to `walkforward-tuning-loop` design); revalidate when it lands.
